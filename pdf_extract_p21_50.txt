Sebagai contoh, catatan "Pemeriksaan USG 1", "Konfirmasi Bunting", dan 
"Catatan Kelahiran" semua terkelompok ra pi sebagai entitas Checkup di bawah

Signifikansi dari kelas ActivityLog tidak dapat diremehkan. Kelas ini 
berfungsi sebagai pencatat jejak audit (audit trail) universal. Ia memiliki relasi 
many  to one ke setiap kelas aktor (Admin, Peternak, Petugas). Desain ini 
menyediakan mekanisme perekaman histori yang komprehensif, merekam 'siapa 
melakukan apa dan kapan' untuk setiap tindakan penting dalam sistem. Hal ini 
fundamental untuk akuntabilitas, resolusi p erselisihan, dan pemeliharaan 
integritas data. Dengan menghubungkannya ke semua kelas aktor, sistem 
memastikan bahwa setiap klik dan perubahan data yang krusial  mulai dari 
pembuatan Request oleh Peternak, validasi oleh Admin, hingga input Checkup 
oleh Petugas  dapat dilacak kembali ke sumbernya.  
Dengan demikian, rancangan diagram kelas ini membentuk arsitektur dan

4.1.3 Pengembangan Prototipe   
Sistem informasi ini dikembangkan menggunakan arsitektur client -server  
yang memisahkan antara antarmuka pengguna ( client -side) dengan logika bisnis 
dan basis data ( server -side). Kedua sisi ini berkomunikasi melalui Application 
Programming Interface (API) dengan protokol REST ( Representational State 
Transfer ). Arsitektur ini memungkinkan pengembangan yang modular, di mana 
sisi front  end dan back  end dapat dikembangkan, diuji, dan diperbarui secara 
independen, serta mendukung skalabilitas dan pemeliharaan sis tem dalam 
jangka panjang.  
Pada sisi frontend , aplikasi bertindak sebagai klien yang bertanggung 
jawab untuk menampilkan data dan menangkap interaksi pengguna (Admin dan 
Petugas). Antarmuka pengguna dikembangkan menggunakan React dan Vite. 
React dipilih karena merupakan pustaka JavaScript yang popule r dan efisien 
untuk membangun antarmuka pengguna yang interaktif dan berbasis komponen, 
serta memiliki ekosistem yang luas. Sementara itu, Vite digunakan sebagai build 
tool karena kemampuannya dalam menyediakan proses pengembangan yang 
cepat dan waktu kompilasi yang minimal, sehingga sangat mendukung 
produktivitas developer.  
Di sisi backend , server mengelola logika bisnis, memproses permintaan 
dari client, berinteraksi dengan basis data, dan mengirimkan data dalam format 
JSON. Untuk membangun REST API, digunakan Node.js, sebuah runtime  
JavaScript yang ringan dan efisien, terutama cocok untuk aplikasi berbasis I/O 


--- PAGE 35 ---
23 
yang intensif seperti layanan web. Node.js juga memiliki ekosistem modul yang 
luas, sehingga mempercepat pengembangan.  
Untuk basis data, digunakan PostgreSQL, sebuah sistem manajemen basis 
data relasional open  source  yang kuat dan andal. PostgreSQL dipilih karena 
kemampuannya dalam menangani query  kompleks, serta skalabilitas dan 
kestabilan dalam pengelolaan data berskala besar.  
Alur kerja sistem, sebagaimana diilustrasikan pada Gambar 12, dimulai 
dari pengguna yang berinteraksi dengan aplikasi web. Setiap aksi pengguna 
menghasilkan permintaan ( request ) yang dikirim ke REST API. REST API 
kemudian memproses logika bisnis dan berinteraksi dengan PostgreSQL untuk 
mengambil atau menyimpan data, lalu mengembalikan respon  ke aplikasi web 
untuk ditampilkan kepada pengguna.

atau kejadian interupsi seperti keguguran. Use case  laporkan hasil kelahiran atau 
keguguran mendeskripsikan bagaimana petugas melaporkan hasil akhir dari proses 
reproduksi setelah melakukan verifikasi di lapangan. Laporan ini akan secara 
definitif menyelesaikan siklus permintaan dalam sistem, baik dengan status sukses 
maupun gagal. Alur kerja untuk pelaporan akhir ini dije laskan pada Tabel 16.
